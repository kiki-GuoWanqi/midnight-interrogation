# 午夜审讯 · Midnight Interrogation

> 一款 AI 驱动的推理审讯游戏。每个嫌疑人都在说谎，你的任务是找出那个说谎最深的人。

🔗 **在线体验：** https://midnight-interrogation.pages.dev

---

## 这款游戏的不同之处

市面上大多数推理游戏是"线性解谜"或"选项对话"——你只能从预设的线索中拼凑真相，无法自由提问，缺乏真正的审讯互动感。

**《午夜审讯》不同：**

- **自由对话审讯** — 你可以向嫌疑人问任何问题，AI 即时生成符合角色人设的回答。没有预设选项，没有固定剧本。
- **每个嫌疑人都会说谎** — 真凶要掩盖罪行，无辜者也有自己的秘密要隐瞒。你需要在三人的矛盾说法中交叉比对、找出漏洞。
- **每局都是新游戏** — 凶手随机指定，AI 动态生成线索和叙事，同一个案件玩三遍体验各不相同。
- **完整故事闭环** — 游戏结束不是只给一个"对或错"，而是还原完整案件经过，逐条解读你接触过的每一条线索。

这不是一个"选择 A 或 B"的推理游戏，而是一个真正的审讯模拟器。

## 产品概述

《午夜审讯》是一款基于 AI 对话的推理游戏。玩家扮演警探，对三名嫌疑人进行自由审讯，通过 25 次提问找出真凶。

每个案件包含三个固定嫌疑人，但**凶手由系统每局随机指定**——这意味着同一个案件，不同对局中真凶不同，每个人的 System Prompt（包括动机、谎言细节、不在场证明）也随之动态变化，每局体验都不相同。

### 当前案件库

| 案件 | 场景 | 嫌疑人 |
|------|------|--------|
| 顶楼的红酒 | CEO 在书房中毒身亡 | 妻子、合伙人、私人秘书 |
| 画廊深夜 | 知名画家死于画廊 | 经纪人、前助理、大藏家 |
| 别墅除夕 | 集团董事长除夕坠楼 | 长子、继女、家庭医生 |

## AI 在产品中的运用

### 1. Prompt Engineering — 角色扮演驱动的审讯体验

每个嫌疑人有两套 System Prompt（无辜版 / 凶手版），根据当局凶手身份动态组合。关键设计：

- **行为规则**统一约束所有角色：不主动暴露、敏感词触发 [紧张]/[慌乱] 情绪标记、保持人设一致
- **无辜者**有自己的秘密要隐瞒（不一定与案件直接相关），会显得可疑
- **凶手**有完整的作案动机、手法、时间线，需要编造不在场证明，被戳穿时会慌乱
- **三人回答相互指向**——无论谁是凶手，其他两人的陈述会自然地把嫌疑引向不同方向，形成推理的"三角验证"

### 2. AI 动态内容生成

- **案件线索 AI 化**：每局 5 条公开线索 + 3 条刑侦支援线索由 AI 根据实际凶手动态生成，2 条暗指真凶、2 条指向无辜者、1 条中性，线索内容随凶手不同而变化
- **结局叙事生成**：游戏结束后，AI 根据玩家实际遭遇的案件走向，生成完整的案件全貌叙事和逐条线索解读，形成故事闭环

## 产品设计关键决策

### 为什么固定案件 + 随机凶手，而不是全动态生成？

全由 AI 生成会带来故事逻辑漏洞、角色矛盾、线索不闭环的问题。固定案件 + 随机凶手在"可控叙事质量"和"可重玩性"之间取得平衡：案件框架是精修过的，确保推理逻辑自洽；凶手随机 + System Prompt 联动 + AI 线索生成保证每局新鲜感。

### 为什么限制 25 次共享提问池？

创造"资源管理"的决策维度——玩家需要思考问谁、问什么、何时使用刑侦支援，而不是无限制地穷举提问。

### 为什么每个无辜者也有秘密？

让所有嫌疑人都"看起来可疑"，避免"谁说话最奇怪谁就是凶手"的简单推理，逼玩家通过交叉对比三人的陈述来寻找矛盾。

## 技术栈

| 层 | 选型 |
|------|------|
| 前端框架 | React 18 |
| 构建工具 | Vite 5 |
| 状态管理 | useReducer（零外部依赖） |
| 样式 | CSS Custom Properties + Noir 主题 |
| AI 模型 | DeepSeek Chat API |
| 后端 | Cloudflare Workers（API 代理，隐藏 API Key） |
| 前端托管 | Cloudflare Pages |
| 外部依赖 | 仅 React + ReactDOM |

### 技术亮点

- **零 UI 库**：从头手写 Noir 视觉风格，完整 CSS 变量系统
- **打字机效果**：逐字渲染 + BGM 同步播放
- **对话状态管理**：useReducer 驱动多嫌疑人对话历史、情绪状态、共享提问池
- **API 安全**：后端 Worker 代理，API Key 不暴露到客户端
- **AI 请求分路**：审讯对话用 300 token 低延迟通道，叙事生成用 2048 token 高容量通道

## 本地运行

```bash
# 1. 安装依赖
npm install

# 2. 启动开发服务器
npm run dev

# 3. 构建生产版本
npm run build
```

> 本地开发时，AI 对话会 fallback 到 mock 数据。需要完整 AI 功能需配置 Cloudflare Worker 并设置 `DEEPSEEK_API_KEY` 环境变量。

## 关于此项目的开发方式

本项目使用 **Vibe Coding** 方法开发——通过 AI 对话驱动开发过程，而非传统的手写代码。核心流程：

1. 用自然语言描述产品想法
2. AI 生成设计文档、技术方案、实现代码
3. 人负责测试、反馈、决策
4. 循环迭代直到产品完成
